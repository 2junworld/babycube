## 1. 프로젝트 한눈에 보기

- **무엇**: 이유식 재고·식단·급여 기록 관리 PWA "베이비큐브"
- **스택**: Vite + React, Firebase(Google 로그인 + Firestore 가족 공유), Tailwind, lucide-react, recharts, vite-plugin-pwa
- **코드 구조** (C-2 1단계 분리 완료, 2026-07-17):
  - `src/theme.js` — 색상 팔레트(C)·카테고리 상수·공용 스타일
  - `src/lib/dates.js` — 날짜·포맷·uid 유틸 (순수 함수)
  - `src/data/nutrition.js` — 시드 재료·성장 단계·영양 태그·궁합 규칙 (정적 데이터)
  - `src/state/appState.js` — seedState·migrateState·reducer + 재고/기록/계산 헬퍼 (UI 없는 로직 계층)
  - `src/store.js` — Store 컨텍스트·useStore
  - `src/components/` — 공용 UI: `common.jsx`(배지·리스트·모달 등 19종), `hints.jsx`(성장·궁합·재고 도움말), `pickers.jsx`(재료·끼니·복사 선택기), `planEditor.jsx`(계획 재료 편집기)
  - `src/lib/` — `dates.js`, `pairing.js`(궁합 계산), `stockAlerts.js`(소진 예측), `stats.js`(급여 통계), `mealLabels.js`, `exporters.js`
  - `src/screens/` — 화면별 파일: `TodayTab` `PlanTab` `StockTab` `RecordTab` `MoreTab` `FeedingLogScreen` `Shell`(탭바·라우팅) `uiPrefs`(뷰 설정 저장)
  - `src/sync/providers.jsx` — 로그인(AuthGate)·가족 설정·FamilyStoreProvider(Firestore 동기화)·DemoProvider·DemoFamilyFlow
  - `src/pwa.jsx` — PwaUpdateProvider/usePwaUpdate/UpdateBanner (PWA 업데이트 감지·적용, App.jsx 최상단에서 제공)
  - `src/App.jsx` — 진입점 + ErrorBoundary + PwaUpdateProvider (약 50줄)
  - Firebase 초기화는 `src/firebase.js`
- **배포**: Vercel ↔ GitHub 연동 완료. **main에 머지하면 자동으로 프로덕션 배포됨** (별도 배포 명령 불필요). 프로젝트: `2junworlds-projects/babycube`
- **작업 방식(관례)**: `claude/babycube-app-dev-*` 브랜치에서 작업 → PR 생성 → main 머지. 사용자가 배포까지 원하는 경우가 대부분이라 머지 = 배포로 이해하면 됨
- **문서**: 배포 절차 상세는 `배포_공유_가이드.md` 참고

## 2. 데모 모드 (개발·검증·스크린샷용)

- URL에 `?demo`를 붙이면 **로그인 없이** 데모 데이터로 앱 전체를 사용 가능 (`DemoProvider`, Firebase 완전 미사용, 새로고침하면 초기화)

- 검증 루틴(이번 세션에서 반복 사용):
  1. `npm install` 후 `npx vite --port 5199`
  2. Playwright(전역 설치됨, 크로미엄: `/opt/pw-browsers/chromium-1194/chrome-linux/chrome`)로 `http://localhost:5199/?demo` 열어 모바일 뷰포트(390×844)에서 UI 조작·스크린샷
  3. 특정 시나리오가 필요하면 `demoState()`를 임시 수정해 재현 → 검증 후 되돌리기

- **PR C**(기록·계획 연동): 급여 기록·식단표 편집 화면에 "+ 시판 제품 추가"(전용 `ProductPicker` — 이미 복잡한 `IngredientPicker`를 건드리지 않기 위해 별도 컴포넌트로 분리), 재고 ON 시 저장 시점 팩 차감/복원, 계획 대비 실제 비교 화면에 제품 항목 표시
- **PR D**(통계·CSV·데모): 월간 리포트에 "제작 vs 시판(끼니 수 기준)" 통계, CSV 내보내기에 시판 컬럼 3개(제품명/팩수/환산g) 추가, 데모 데이터에 제품 2종(재고 있음/소진) + 관련 급여 기록·활동 로그 추가(`productStockEnabled` 기본 ON으로 데모 체험 가능)
- **PR E**(후속: 인라인 등록·궁합·유제품·재고 통합): `ProductPicker`에 "새 시판 제품 등록" 인라인 생성(만들고 바로 선택). 시판 제품 궁합을 혼합 큐브와 동일한 방식(포함 재료 태그 합산)으로 계산해 제품 상세에 표시 + 급여기록/식단표 궁합 추천 계산에도 반영(제품 자체는 재료 마스터에 등록하지 않는 전용 계산 — `pairingNamesOf`가 제품 항목을 포함 재료 이름으로 펼침). 재료 카테고리에 **유제품** 추가. 재고 탭은 시판 제품을 상단 고정 섹션 대신 재료와 같은 목록에서 카테고리처럼 통합 표시(+"시판" 필터 칩), 재료 정보 탭도 세그먼트 대신 "시판"을 카테고리 필터 중 하나로 통합(먹어본 여부 필터는 제품엔 미적용)
- **PR F**(후속: 제공량 override + 카드 정보): 바로기록·끼니 계획에서 시판 제품의 제공량(g)을 직접 입력 가능하게(`gramsOverride`) — 한 팩을 다 안 먹인 경우 대응. 제품 카드(피커·편집 화면·재료 정보 목록)에 포함 재료를 칩으로 바로 노출
- **PR G**(후속: 입력 UI 통일): 제공량 입력 방식을 링크형 토글 대신 냉동/냉장·큐브/그램과 동일한 `Segmented` 컴포넌트("팩으로 입력"/"중량으로 입력")로 통일
- **버그 수정**(PR H): PR G 방식에서 팩 수(qty)와 gramsOverride가 서로 독립적으로 남아 모드 전환 시 어긋나던 문제 수정 — 중량 모드에서는 팩 입력칸을 아예 숨기고, 입력한 g을 담는 데 필요한 최소 팩 수(올림, `Math.ceil`)를 자동 계산해 재고 차감에 사용. 재료의 큐브/그램 토글과 완전히 동일한 방식으로 통일
- **스코프 제외**(제안서에도 추후 과제로 명시): 성분표 사진 인식, 장보기 리스트 자동 등록, 유통기한 알림 배너, 여행 모드 시판 반영

### PWA 설치형 앱 전환 (2026-07-19, v1.5.1)

외부 제안서(베이비큐브 기능 제안서 — PWA 설치형 앱 전환) 기반. 기존에 이미 `vite-plugin-pwa`(manifest·서비스 워커)가 붙어 있었지만 `registerType:'autoUpdate'` + 아이콘이 SVG 하나뿐이라 iOS 홈 화면 추가 시 문제(투명 배경 아이콘, safe-area 미대응)와 "배포 후 앱을 완전히 껐다 켜야 반영되는" 업데이트 불편이 있었음 — 이를 해소.

- **아이콘 세트**: 기존 `pwa-icon.svg`(둥근 사각형, favicon용 그대로 유지)에서 Playwright로 PNG 렌더링해 `pwa-192.png`/`pwa-512.png`(둥근 배경, 일반 용도)와 `pwa-512-maskable.png`/`apple-touch-icon.png`(각진 풀블리드 배경 — OS가 자체 마스킹/라운딩하므로 소스는 각지게, 아이콘 도형은 이미 중앙 62.5% 안이라 80% 세이프존 만족) 생성. sharp 등 이미지 툴이 환경에 없어 Chromium 스크린샷으로 대체 생성(재생성 시 동일 방식 사용 가능)
- **vite.config.js**: `registerType:'prompt'` + `injectRegister:null`(자동 등록 스크립트 비활성화, `useRegisterSW` 훅으로 직접 등록) + `runtimeCaching:[]`(Firebase 요청은 서비스 워커가 절대 캐시하지 않음을 명시) + globPatterns에 png 추가
- **`src/pwa.jsx`**(신규): `PwaUpdateProvider`(App.jsx 최상단에서 1회 마운트, 1시간마다 자동 새 버전 확인) + `usePwaUpdate()` 훅 + `UpdateBanner`(새 버전 있을 때만 상단에 "새 버전이 있어요 · 업데이트" 배너, 탭하면 적용). `FamilyStoreProvider`에서는 적용 전 진행 중인 Firestore 저장(`pendingRef`)이 끝나길 기다린 뒤 새로고침 — 저장 도중 새로고침으로 요청이 끊기는 것 방지
- **`MoreTab`**: 버전 표기 옆에 "업데이트 확인" 수동 버튼 추가(새 버전이 있으면 버튼 대신 안내 문구로 전환), "앱을 완전히 종료했다 열면 적용" 문구 삭제
- **`index.html`**: `apple-touch-icon`·`apple-mobile-web-app-*` 메타 태그, `viewport-fit=cover` 추가
- **safe-area**: 공용 `BottomSheet` 컴포넌트와 손으로 작성한 하단 시트 6곳(BatchModal·ProductLotModal·IntroEditModal·MealSlotEditModal·IngredientPicker·DisplayNameSheet)에 `env(safe-area-inset-bottom)` 하단 패딩 추가 — iOS standalone에서 홈 인디케이터와 겹치지 않게 함(브라우저 탭에서는 0px라 기존 화면과 동일)
- **iOS 구글 로그인 폴백**: standalone(홈 화면 설치) 모드에서는 `signInWithPopup` 대신 처음부터 `signInWithRedirect` 사용(팝업이 조용히 멈추는 known issue 대응), 일반 브라우저에서도 `auth/popup-blocked` 등 팝업 실패 시 리다이렉트로 재시도 + `getRedirectResult`로 리다이렉트 완료 후 에러 표시
- **검증 범위**: `npx vite build` → dist에 manifest.webmanifest·sw.js·아이콘 생성 확인, `vite preview` + Playwright로 서비스 워커가 실제로 register→active 상태가 되는 것과 manifest/아이콘 200 응답 확인, 데모 모드 전체 탭 회귀 스모크 테스트(콘솔 에러 없음)
- **⚠️ 미검증(실기기 필요, 제안서에도 명시된 리스크)**: 안드로이드/아이폰 실기기 설치("홈 화면에 추가") 후 standalone 실행·아이콘·safe-area, 그리고 **iOS standalone에서의 구글 로그인(가장 리스크가 큰 지점)**은 이 개발 환경(샌드박스, 실기기 없음)에서 확인 불가 — 배포 후 실기기로 직접 확인 필요
- **후속 과제**(범위 아님, 제안서 3장): Firestore 오프라인 캐시, 웹 푸시 알림, iOS 기종별 스플래시 이미지

## 5. 데이터 모델 메모 (이번 세션에서 추가된 필드)

| 위치 | 필드 | 의미 |
|---|---|---|
| plans[date][].meal | `fromRecord?: true` | 급여 기록에서 생성/동기화된 끼니 (배지 표시용) |
| logs[date][].items[] | `deductedQty` | 실제 재고에서 차감된 양 (냉동=큐브, 냉장=g). 없으면 qty로 폴백 |
| logs[date][] | `time` | 실제 급여 시간 (계획 시간과 별개) |
| logs[date][] | `planSnapshot` | 저장 시점 계획 스냅샷 (계획 대비 비교용, 바로 기록은 null) |
| ingredients[name] | `staple?: true` | 주식(밥·죽) 재료 - 장보기 자동 등록·여행 계산 제외 (재료 정보에서 토글) |
| logs[date][].items[] | `deductedBatches: [{batchId, qty}]` | 배치별 차감 내역 (C-3) - 수정·삭제 시 원 배치로 정확히 복원. 없으면 최근 배치 복원 폴백 |
| state | `memberProfiles: {[uid]: {name, color, joinedAt}}` | 구성원 표시명·뱃지 색상 (작성자 추적). 없으면 로그인 시 표시명 입력 바텀시트로 채움 |
| state | `activity: [{id, at, by, action, kind, summary, ref?}]` | 활동 로그, 최근 200건 순환 (작성자 추적) |
| logs·plans[].meal·stock[].batches[] | `createdBy/createdAt`, `updatedBy/updatedAt` | 작성자 메타 (작성자 추적). 구버전 데이터는 없음 - 뱃지 생략, 소급 기입 안 함 |
| state | `products: {[id]: {id,name,brand,packG,ingredients,memo,createdBy/At,updatedBy/At}}` | 시판 제품 마스터 (시판 이유식) |
| state | `productStock: {[productId]: {lots:[{id,buyDate,exp,packs,createdBy/At,updatedBy/At}]}}` | 시판 제품 재고(팩 단위 lot). 토글 OFF여도 데이터는 유지(숨김만) |
| settings | `productStockEnabled: boolean` (기본 false) | 시판 재고관리 전역 토글 - 제품 관리 자체는 토글과 무관하게 항상 가능 |
| logs·plans[].meal 항목 | `{source:"product", productId, productName, packG, qty, gramsOverride?, deduct, deductedQty?, deductedLots?}` | 시판 제품 급여/계획 항목. `qty`=재고 차감 단위(팩, 항상 정수), `gramsOverride`는 실제 제공량(g) override - null이면 `qty*packG` 그대로 사용 |

마이그레이션은 `migrateState()`에서 처리. 새 필드는 모두 "없으면 폴백" 방식이라 기존 데이터와 호환됨.

## 6. 알려진 엣지 케이스 / 다음 작업 후보

- 같은 이름(예: 점심)의 기록이 하루에 2개면: 오늘 탭에는 둘 다 보이지만, 급여표 상세 비교 화면은 첫 기록 기준 (권장하지 않는 사용 패턴)
- **개선 후보(제안만 됨)**: 바로 기록에서 식단표에 있는 끼니 이름을 고르면 계획된 재료를 미리 채워주기
- `restoreFrozen/restoreFridge`는 해당 재료의 배치가 모두 삭제된 경우 복원 불가 (기존 한계)
- 소개 이미지 문구/구성 변경 요청 가능성 있음 (3절 참고)

## 7. 이어서 작업하는 방법

```bash
npm install          # 의존성
npx vite             # 개발 서버 (?demo 붙여서 로그인 없이 확인)
npx vite build       # 커밋 전 빌드 확인 (이번 세션 관례)
```

- 새 작업은 main에서 `claude/babycube-app-dev-*` 브랜치를 따서 진행 → PR → 머지(=배포)
- UI 변경은 데모 모드 + Playwright로 실제 동작을 확인한 뒤 커밋하는 것을 관례로 함
- PWA라서 배포 후 사용자 기기에는 앱 완전 종료 후 재실행해야 반영됨
