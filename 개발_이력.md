## 1. 프로젝트 한눈에 보기

- **무엇**: 이유식 재고·식단·급여 기록 관리 PWA "베이비큐브"
- **스택**: Vite + React, Firebase(Google 로그인 + Firestore 가족 공유), Tailwind, lucide-react, recharts, vite-plugin-pwa
- **코드 구조** (C-2 1단계 분리 완료, 2026-07-17):
  - `src/theme.js` — 색상 팔레트(C)·카테고리 상수·공용 스타일
  - `src/lib/dates.js` — 날짜·포맷·uid 유틸 (순수 함수)
  - `src/data/nutrition.js` — 시드 재료·성장 단계·영양 태그·궁합 규칙 (정적 데이터)
  - `src/state/appState.js` — seedState·migrateState·reducer + 재고/기록/계산 헬퍼 (UI 없는 로직 계층)
  - `src/store.js` — Store 컨텍스트·useStore
  - `src/components/` — 공용 UI: `common.jsx`(배지·리스트·모달 등 19종), `hints.jsx`(성장·궁합·재고 도움말), `pickers.jsx`(재료·끼니·복사 선택기), `planEditor.jsx`(계획 재료 편집기)
  - `src/lib/` — `dates.js`, `pairing.js`(궁합 계산), `stockAlerts.js`(소진 예측), `stats.js`(급여 통계), `mealLabels.js`, `exporters.js`
  - `src/screens/` — 화면별 파일: `TodayTab` `PlanTab` `StockTab` `RecordTab` `MoreTab` `FeedingLogScreen` `Shell`(탭바·라우팅) `uiPrefs`(뷰 설정 저장)
  - `src/sync/providers.jsx` — 로그인(AuthGate)·가족 설정·FamilyStoreProvider(Firestore 동기화)·DemoProvider·DemoFamilyFlow
  - `src/pwa.jsx` — PwaUpdateProvider/usePwaUpdate/UpdateBanner (PWA 업데이트 감지·적용, App.jsx 최상단에서 제공)
  - `src/App.jsx` — 진입점 + ErrorBoundary + PwaUpdateProvider (약 50줄)
  - Firebase 초기화는 `src/firebase.js`
- **배포**: Vercel ↔ GitHub 연동 완료. **main에 머지하면 자동으로 프로덕션 배포됨** (별도 배포 명령 불필요). 프로젝트: `2junworlds-projects/babycube`
- **작업 방식(관례)**: `claude/babycube-app-dev-*` 브랜치에서 작업 → PR 생성 → main 머지. 사용자가 배포까지 원하는 경우가 대부분이라 머지 = 배포로 이해하면 됨
- **문서**: 배포 절차 상세는 `배포_공유_가이드.md` 참고

- URL에 `?demo`를 붙이면 **로그인 없이** 데모 데이터로 앱 전체를 사용 가능 (`DemoProvider`, Firebase 완전 미사용, 새로고침하면 초기화)

- 검증 루틴(이번 세션에서 반복 사용):
  1. `npm install` 후 `npx vite --port 5199`

- **PR E**(후속: 인라인 등록·궁합·유제품·재고 통합): `ProductPicker`에 "새 시판 제품 등록" 인라인 생성(만들고 바로 선택). 시판 제품 궁합을 혼합 큐브와 동일한 방식(포함 재료 태그 합산)으로 계산해 제품 상세에 표시 + 급여기록/식단표 궁합 추천 계산에도 반영(제품 자체는 재료 마스터에 등록하지 않는 전용 계산 — `pairingNamesOf`가 제품 항목을 포함 재료 이름으로 펼침). 재료 카테고리에 **유제품** 추가. 재고 탭은 시판 제품을 상단 고정 섹션 대신 재료와 같은 목록에서 카테고리처럼 통합 표시(+"시판" 필터 칩), 재료 정보 탭도 세그먼트 대신 "시판"을 카테고리 필터 중 하나로 통합(먹어본 여부 필터는 제품엔 미적용)
- **PR F**(후속: 제공량 override + 카드 정보): 바로기록·끼니 계획에서 시판 제품의 제공량(g)을 직접 입력 가능하게(`gramsOverride`) — 한 팩을 다 안 먹인 경우 대응. 제품 카드(피커·편집 화면·재료 정보 목록)에 포함 재료를 칩으로 바로 노출
- **PR G**(후속: 입력 UI 통일): 제공량 입력 방식을 링크형 토글 대신 냉동/냉장·큐브/그램과 동일한 `Segmented` 컴포넌트("팩으로 입력"/"중량으로 입력")로 통일
- **버그 수정**(PR H): PR G 방식에서 팩 수(qty)와 gramsOverride가 서로 독립적으로 남아 모드 전환 시 어긋나던 문제 수정 — 중량 모드에서는 팩 입력칸을 아예 숨기고, 입력한 g을 담는 데 필요한 최소 팩 수(올림, `Math.ceil`)를 자동 계산해 재고 차감에 사용. 재료의 큐브/그램 토글과 완전히 동일한 방식으로 통일
- **스코프 제외**(제안서에도 추후 과제로 명시): 성분표 사진 인식, 장보기 리스트 자동 등록, 유통기한 알림 배너, 여행 모드 시판 반영

- **`index.html`**: `apple-touch-icon`·`apple-mobile-web-app-*` 메타 태그, `viewport-fit=cover` 추가
- **safe-area**: 공용 `BottomSheet` 컴포넌트와 손으로 작성한 하단 시트 6곳(BatchModal·ProductLotModal·IntroEditModal·MealSlotEditModal·IngredientPicker·DisplayNameSheet)에 `env(safe-area-inset-bottom)` 하단 패딩 추가 — iOS standalone에서 홈 인디케이터와 겹치지 않게 함(브라우저 탭에서는 0px라 기존 화면과 동일)
- **구글 로그인 방식**: 최초 배포 시엔 standalone(홈 화면 설치) 모드에서 `signInWithRedirect`를 우선 사용하도록 했으나, 실기기 검증 결과 **안드로이드·iOS standalone 모두에서 리다이렉트가 구글에서 돌아온 뒤 로그인 상태를 못 받아오고 조용히 로그인 화면으로 되돌아가는 문제**가 확인돼(브라우저 탭에서는 정상 동작) 최종적으로는 플랫폼·standalone 여부와 무관하게 **항상 `signInWithPopup`을 우선 사용**하도록 변경. 팝업이 명백히 막히거나 미지원인 경우(`auth/popup-blocked`, `auth/operation-not-supported-in-this-environment`)에만 리다이렉트로 재시도, `getRedirectResult`로 리다이렉트 완료 후 에러 표시는 유지

### 재료/시판 추가 버튼 통합 + 시판 제품 검색어 즉시등록 (2026-07-20)

기존엔 "재료 추가"·"시판 제품 추가" 버튼이 급여기록·식단표(끼니 편집·여러 날짜 저장) 3곳에 각각 나란히 있었고, 시판 제품은 재료와 달리 검색창에 없는 이름을 입력해도 바로 새로 등록할 수 없었음. 두 이슈를 함께 검토한 뒤(전체 통합안은 로직이 꼬일 리스크가 있어 기각) 아래 방식으로 구현:

- **버튼 통합**: 두 버튼을 "+ 재료 추가" 단일 버튼으로 통합 — `IngredientPicker`가 열리고, 그 안에서 "시판 제품에서 찾기" 링크로 `ProductPicker`로 전환하는 방식. `IngredientPicker`는 9곳에서 재사용되므로 새 prop `onSwitchToProduct`를 **옵션**으로 추가해(값을 안 넘기면 링크 자체가 렌더되지 않음) 급여기록·식단표(끼니 편집/여러 날짜 저장) 3곳에만 적용, 나머지 6곳은 동작 변화 없음
- **양방향 전환 + 검색어 유지**: `ProductPicker`에도 반대 방향 "재료에서 찾기" 링크(`onSwitchToIngredient`) 추가. 전환 시 타이핑 중이던 검색어를 부모의 `switchQuery` state로 들고 가 상대 피커의 `initialQuery`로 그대로 이어줌(각 피커는 전환마다 새로 마운트되므로 `useState(initialQuery)`로 1회만 반영)
- **시판 제품 즉시 등록**: `ProductPicker` 검색창에 입력한 텍스트가 기존 제품과 정확히 일치하지 않으면 "'{검색어}' 새 제품으로 등록" 버튼으로 바뀌고, 탭하면 `ProductEditSheet`가 그 이름으로 미리 채워진 채 열림(`initialName` prop) — 재료 추가와 동일한 사용 흐름
- **영향 범위**: `FeedingLogScreen`, `PlanTab`(`MealEditScreen`·`BulkSaveScreen`)만 변경. `planEditor.jsx`의 별도 `AddProductButton` 컴포넌트는 삭제(단일 버튼 흐름으로 대체돼 더는 필요 없음)
- **검증**: 데모 모드 + Playwright로 3개 진입점 모두에서 단일 버튼 노출, 검색어 유지 양방향 전환, 신규 제품 등록 시 이름 프리필 확인. 전체 탭 회귀 스모크 테스트 통과(콘솔 에러 없음)

### 재료 선택기 개선 6종 + 후속 버그 수정 (2026-07-20)

위 버튼 통합 작업 직후 이어진 요청. `IngredientPicker`/`ProductPicker`(`src/components/pickers.jsx`)에 다음을 반영:

- **정렬 상태 기기 저장**: 정렬 칩(즐겨찾기·오늘 사용 재료 제외·궁합·재고순·카테고리순) 선택을 `localStorage`에 저장해 피커를 다시 열어도 유지
- **카테고리 비율 그래프에 시판 제품 반영**: `appState.js`의 `catTotals`(및 `stats.js`의 `monthStats`)가 시판 제품 항목도 포함시키도록 변경 - 포함 재료가 등록된 제품은 그 재료들의 카테고리에 균등 배분, 미등록 제품은 일반적인 이유식 구성비 추정치(`productCatSplit`, 탄수화물 55%·단백질 20%·채소 20%·과일 3%·유제품 2%)로 배분. 기존 "시판 제품은 함량을 몰라 통계에서 제외" 정책을 대체
- **재료 ↔ 시판 스와이프 전환**: 기존 텍스트 링크를 재료/시판 제품 `Segmented` 탭으로 교체하고, 시트를 좌우로 드래그해도 전환되도록 포인터 이벤트 기반 스와이프 제스처 추가(진입 시 슬라이드인 애니메이션, 전환 불가 방향은 고무줄 저항). 빠른 pointermove 연속 이벤트가 React 리렌더보다 앞서가며 임계값 판정에 오래된 state를 참조하던 레이스 컨디션을 ref로 해결
- **시판 제품 인라인 즉시등록**: `ProductEditSheet`의 폼 부분을 `ProductEditForm`으로 분리해, `ProductPicker`에서 "새 제품으로 등록"을 누르면 별도 화면 전환 없이 같은 시트 안에 등록 폼이 펼쳐지도록 변경(재료 선택기의 "검색어로 바로 새 재료 추가"와 동일한 경험). 이 인라인 폼 내부의 "포함 재료" 하위 피커는 `createPortal`로 `document.body`에 직접 렌더링 - 스와이프용 `transform`이 걸린 조상 안에 중첩되면 `position:fixed`의 컨테이닝 블록이 바뀌어 화면 밖으로 밀려나는 문제가 있어 우회
- **검증**: Playwright로 6개 항목 모두 개별 시나리오 테스트(정보 오버레이 왕복, 선택 재료 정렬, 새로고침 없이 localStorage 유지, 카테고리 분할 계산 단위 테스트 + 실제 화면 확인, 양방향 탭·스와이프, 인라인 등록 폼 저장까지) + 재료 정보 위키의 기존 `ProductEditSheet` 경로 회귀 확인 + 전체 탭 스모크 테스트

### 실기기 피드백 반영: 시트 높이 고정 + 스와이프 인식 개선 (2026-07-20)

### 개선 제안함을 가족별 데이터 → 전체 공유 컬렉션으로 전환 (2026-07-23)

"다른 사용자(다른 가족)가 올린 개선 제안이 안 보인다"는 문제 제보에 대한 근본 수정. 이 앱은 가족(`families/{familyId}`)마다 데이터가 완전히 분리되는 구조라, 개선 제안을 가족별 `state.feedback`으로 저장하는 한 지인이 안내 링크로 별도 가족을 만들어 남긴 제안은 원 사용자에게 구조적으로 절대 보이지 않았음(버그가 아니라 설계상 당연한 결과). 이를 어느 가족이든 함께 보는 전체 공유 컬렉션으로 옮겨 근본적으로 해결.

- **데이터 모델**: 최상위 Firestore 컬렉션 `globalFeedback/{id}: {text, by(uid), byName(작성 당시 표시명 스냅샷), familyId, at}` 신설. 가족별 `state.feedback` 필드·`FEEDBACK_ADD`/`FEEDBACK_DELETE`/`RESTORE_FEEDBACK` 리듀서 액션은 제거하고, `FeedbackScreen`이 `onSnapshot`/`addDoc`/`deleteDoc`으로 이 컬렉션을 직접 구독·기록한다(가족 범위 동기화 파이프라인을 타지 않음). `byName`을 작성 시점에 스냅샷으로 저장하는 이유: 다른 가족의 `memberProfiles`에는 애초에 그 작성자의 uid가 없어 이름을 못 찾기 때문
- **1회성 마이그레이션**: 이 기능 도입 이후 이미 가족별 `state.feedback`에 쌓인 제안이 있는 가족은, 앱 로드 시 `FamilyStoreProvider`가 그 항목들을 `globalFeedback`으로 복사한 뒤 로컬 `state.feedback`을 비움(`FEEDBACK_MIGRATED`). 실패하면 다음 로드 때 재시도(멱등)
- **보안 규칙**(`firestore.rules`): `globalFeedback`은 조회를 로그인 없이도 허용(`allow read: if true`) - 개선 제안 텍스트는 민감한 개인정보가 아니라고 보고, 별도 계정 생성 없이도 Claude Code 세션 등에서 바로 확인할 수 있게 열어둠. 재고·기록·아기 정보가 담긴 `families` 컬렉션은 기존 규칙대로 로그인한 가족 멤버만 접근 가능해 전혀 영향 없음. 작성은 로그인 필수 + 본인 uid로만, 삭제도 작성자 본인만 가능하도록 제한
- **UI**(`FeedbackScreen`): 다른 가족이 남긴 항목은 "다른 가족" 배지로 구분 표시하고, 삭제 버튼은 본인이 작성한 항목에서만 노출. 데모 모드(`?demo`)는 여전히 실제 Firestore를 건드리지 않고 로컬 상태로만 동작(기존 데모 원칙 유지)
- **검증**: 데모 모드에서 제안 작성→로컬 반영→전체 복사(작성자·날짜 포함) 확인 + 전체 탭 회귀 스모크 테스트 통과. 배포·규칙 게시 후 실제 가족 계정이 남긴 제안이 `globalFeedback`에 정상 저장되고, Firestore REST API(`GET .../documents/globalFeedback`, 인증 없이)로 조회되는 것까지 실사용으로 확인. `families` 컬렉션은 동일 요청이 여전히 403으로 막혀 다른 데이터엔 영향 없음도 함께 확인
- **배포 시 유의**: 코드 배포와 별개로, Firebase 콘솔 → Firestore → 규칙 탭에서 갱신된 `firestore.rules` 내용을 다시 게시해야 `globalFeedback` 컬렉션이 실제로 열림(배포_공유_가이드.md 4단계와 동일한 수동 절차)

### 지난 날짜 급여 기록 추가 (개선 제안 반영) (2026-07-23)

전체 공유 개선제안함으로 처음 접수된 실제 제안: "오늘에 대한 급여기록만 가능해요. 지난날짜에 대한 급여기록 추가가 있으면 좋겠어요!" 확인 결과 실제 문제였음 - `FeedingLogScreen`(급여 기록 화면)과 `LOG_SAVE` 리듀서는 애초에 날짜에 무관하게 동작하는데, 이 화면으로 들어가는 진입점이 `TodayTab`(오늘 탭, 항상 `date=오늘`)뿐이라 지난 날짜를 기록할 방법이 실질적으로 없었음.

- **수정**: `FeedingMonthPanel`(기록 탭 → 급여표 → 월별, `RecordTab.jsx`)에 날짜 선택 시 "N월 N일 기록 추가" 버튼을 추가. 미래 날짜는 기록할 대상이 없으므로 버튼을 노출하지 않음(`selected <= todayISO()`)
- **검증**: 월별 급여표에서 기록 없는 과거 날짜 선택 → 기록 추가 버튼 노출 → 재료 선택·저장까지 Playwright로 실제 저장 흐름 확인(그 과거 날짜에 정상 반영, "기록 없음" 문구 사라짐) + 전체 탭 회귀 스모크 테스트 통과
- **후속 개선(오늘 탭과 동일한 화면으로 통일)**: 위 1차 구현은 "기록 추가"를 누르면 바로 빈 급여 기록 화면(`FeedingLogScreen`)이 떴는데, 그 날짜에 이미 계획된 끼니가 있어도 무시되고 항상 계획 없이 새로 입력해야 해서 오늘 탭과 경험이 달랐음(사용자 피드백으로 발견). `TodayTab.jsx`의 끼니 카드 로직(계획↔기록 매칭, 예정/대기/완료 상태, 기록하기·수정 버튼)을 `date`를 매개변수로 받는 `DayFeedingCards` 컴포넌트로 분리해 오늘 탭·새 `DayRecordScreen`(지난 날짜용, `route: "dayRecord"`)이 함께 재사용하도록 통일. 이제 월별 급여표에서 "기록 추가"를 누르면 그 날짜의 계획된 끼니 카드들이 오늘 탭과 똑같이 뜨고, 계획이 없으면 "바로 기록"으로 안내됨
- **검증**: 계획·기록이 이미 있는 과거 날짜 선택 시 오늘 탭과 동일한 카드(완료/예정 상태·수정·기록하기 버튼)가 뜨는지, 계획·기록이 전혀 없는 날짜는 "바로 기록" 안내로 빠지는지, 오늘 탭 자체가 리팩터링 후에도 그대로 렌더되는지 Playwright로 확인 + 전체 탭 회귀 스모크 테스트 통과
- **후속 개선(오늘 탭에서 직접 과거 날짜로 이동)**: 위 두 진입점(기록 탭 경유)과 별개로, 오늘 탭 자체에서도 날짜를 넘나들 수 있게 요청받아 `◀ 날짜 ▶` 내비게이션 행을 헤더 아래에 추가. `DayFeedingCards`가 이미 date를 매개변수로 받도록 분리돼 있어 `TodayTab`에 `viewDate` 로컬 상태만 추가하면 바로 재사용 가능했음. 미래로는 못 가도록(▶ 버튼 비활성화 + `disabled`) 막고, 과거로 이동하면 "오늘로" 버튼이 나타나 바로 복귀 가능. 냉장 소진 임박 배너·곧 떨어지는 재료 알림은 "오늘~내일" 기준의 실시간 재고 안내라 과거 날짜를 보는 동안은 혼란을 줄 수 있어 오늘(`isToday`)일 때만 노출
- **검증**: ◀ 버튼으로 여러 날짜 이동 후 헤더·날짜 표시가 바뀌는지, 과거 날짜에서 재고 알림 배너들이 숨겨지는지, ▶ 버튼이 오늘에서 비활성 상태인지, "오늘로" 버튼으로 정상 복귀하는지 Playwright로 확인 + 전체 탭 회귀 스모크 테스트 통과

### 오늘 탭 날짜 이동 UX 다듬기 + 급여표 미기록 표시 + 계획 대비 % (2026-07-23)

위 날짜 이동 기능에 대한 후속 피드백 5건 반영.

- **날짜 내비게이션 균형 수정**: `◀ 날짜 ▶` 행이 `justify-center` + 가변 폭의 "오늘로" 버튼 조합이라, 그 버튼이 나타나는 순간 가운데 날짜 텍스트가 오른쪽으로 밀려 보였음. `grid-template-columns: 1fr auto 1fr`로 좌(◀)/중(날짜)/우(▶·오늘로·바로가기) 3열로 재구성해, 우측 클러스터 내용물과 무관하게 날짜가 항상 화면 정중앙에 오도록 수정
- **하단 탭 라벨에 보고 있는 날짜 표시**: 오늘 탭에서 과거 날짜로 이동하면 하단 탭바의 "오늘" 글자가 그 날짜(예: `07-20`)로 바뀌어 지금 뭘 보고 있는지 한눈에 알 수 있음. `viewDate` 상태를 `TodayTab` 로컬에서 `Shell`로 끌어올려 하단 탭 라벨 렌더링에서도 참조할 수 있게 하고, 다른 탭을 누르면 자동으로 오늘로 리셋됨
- **급여표 주별에도 기록 추가 진입점**: 지금까지 월별 화면에만 있던 "기록 추가"를 주별 그리드에도 추가. 기록이 없는 칸 중 계획이 있었는데 아직 기록 안 한 지난 끼니는 "미기록" 배지(주황)로 눈에 띄게, 계획도 기록도 없는 칸은 옅은 + 아이콘으로 표시하고 탭하면 `DayRecordScreen`으로 이동(오늘 날짜 칸도 + 로 추가 가능하되 "미기록"으로는 표시 안 함 - 아직 하루가 안 끝났으므로)
- **급여표 월별 달력에도 미기록 표시**: 계획은 있었는데 하나라도 기록이 안 된 지난 날짜의 달력 칸에 작은 주황 점 표시(범례 포함)
- **월별 달력 표시 전환(섭취량 g / 계획 대비 %)**: 달력 각 칸에 그램 수 대신 "그 날 계획 총량 대비 실제 섭취량 비율"도 볼 수 있도록 토글 추가. 계획이 없는 날은 비교 대상이 없어 "-"로 표시
- **오늘 탭 → 기록 탭 월별 바로가기**: 오늘 탭 날짜 이동 행에 캘린더 아이콘 버튼 추가 - 누르면 보고 있던 날짜가 그대로 선택된 채로 기록 탭 → 급여표 → 월별 화면으로 바로 이동(`UI_STATE.recordMonthSelected`로 전달). 별도 캘린더 팝업을 새로 만드는 대신 이미 있는 월별 화면을 재사용하는 더 가벼운 방식을 선택
- **검증**: 날짜 이동 시 날짜 텍스트가 항상 가운데 오는지, 하단 탭 라벨이 날짜로 바뀌고 다른 탭 이동 시 리셋되는지, 오늘 탭에서 바로가기를 눌렀을 때 보던 날짜가 그대로 선택된 월별 화면이 뜨는지, 실제로 급여 기록을 하나 삭제해 그 칸이 (오늘이면 +, 과거면 미기록으로) 올바르게 갈리는지 Playwright로 확인 + 전체 탭 회귀 스모크 테스트 통과

## 5. 데이터 모델 메모 (이번 세션에서 추가된 필드)

| state | `activity: [{id, at, by, action, kind, summary, ref?}]` | 활동 로그, 최근 200건 순환 (작성자 추적) |

| state | `categories: [{id, name, color, light}]` | 재료 분류 카테고리 (더보기 → 카테고리 관리에서 추가/수정/삭제). 기본 5개는 이전 `theme.js` 고정값을 그대로 시드 |

| state | `productStock: {[productId]: {lots:[{id,buyDate,exp,packs,createdBy/At,updatedBy/At}]}}` | 시판 제품 재고(팩 단위 lot). 토글 OFF여도 데이터는 유지(숨김만) |

## 6. 알려진 엣지 케이스 / 다음 작업 후보

- 같은 이름(예: 점심)의 기록이 하루에 2개면: 오늘 탭에는 둘 다 보이지만, 급여표 상세 비교 화면은 첫 기록 기준 (권장하지 않는 사용 패턴)

- `restoreFrozen/restoreFridge`는 해당 재료의 배치가 모두 삭제된 경우 복원 불가 (기존 한계)
