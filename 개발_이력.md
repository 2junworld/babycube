# 베이비큐브 — 개발 이력 및 이어서 작업 가이드

> 새 세션(코워크 등)에서 이어서 작업할 때 이 문서를 먼저 읽어주세요.
> 마지막 갱신: 2026-07-15 (PR #5까지 반영)

## 1. 프로젝트 한눈에 보기

- **무엇**: 이유식 재고·식단·급여 기록 관리 PWA "베이비큐브"
- **스택**: Vite + React, Firebase(Google 로그인 + Firestore 가족 공유), Tailwind, lucide-react, recharts, vite-plugin-pwa
- **코드 구조**: 앱 전체가 `src/App.jsx` 단일 파일(약 5,000줄). `seedState()`(초기 상태) → `reducer`(모든 상태 변경) → 컴포넌트 순으로 배치. Firebase 초기화는 `src/firebase.js`
- **배포**: Vercel ↔ GitHub 연동 완료. **main에 머지하면 자동으로 프로덕션 배포됨** (별도 배포 명령 불필요). 프로젝트: `2junworlds-projects/babycube`
- **작업 방식(관례)**: `claude/babycube-app-dev-*` 브랜치에서 작업 → PR 생성 → main 머지. 사용자가 배포까지 원하는 경우가 대부분이라 머지 = 배포로 이해하면 됨
- **문서**: 배포 절차 상세는 `배포_공유_가이드.md` 참고

## 2. 데모 모드 (개발·검증·스크린샷용)

- URL에 `?demo`를 붙이면 **로그인 없이** 데모 데이터로 앱 전체를 사용 가능 (`DemoProvider`, Firebase 완전 미사용, 새로고침하면 초기화)
- 데모 데이터는 `demoState()`: 재고 8종, 오늘 식단 3끼(아침은 기록 완료), 과거 13일치 급여 기록
- 검증 루틴(이번 세션에서 반복 사용):
  1. `npm install` 후 `npx vite --port 5199`
  2. Playwright(전역 설치됨, 크로미엄: `/opt/pw-browsers/chromium-1194/chrome-linux/chrome`)로 `http://localhost:5199/?demo` 열어 모바일 뷰포트(390×844)에서 UI 조작·스크린샷
  3. 특정 시나리오가 필요하면 `demoState()`를 임시 수정해 재현 → 검증 후 되돌리기

## 3. 스토어 소개 이미지

- 위치: `store-assets/app-screens/01_today.png` ~ `05_ingredient.png` (1080×1920, 스토어 스크린샷 규격)

- 구버전 일러스트형 이미지는 `store-assets/01_overview.png` 등으로 남아 있음 (별개)

## 4. 이번 세션 작업 이력 (시간순)

모두 main에 머지 + 배포 완료.

### PR #1 — 소개 이미지 + 데모 모드 + 오늘 탭 표시 버그
- 소개 이미지 5장, `?demo` 데모 모드 추가
- **버그 수정**: 오늘 탭의 기록 완료된 끼니 카드가 실제 급여한 재료가 아니라 식단표(계획) 재료를 보여주던 문제 → 완료 카드는 기록(`log.items`) 기준으로 표시
- 급여 기록 저장 시 재고 차감 내역 토스트 추가, 실행취소 없는 토스트는 실행취소 버튼 숨김

### PR #2 — 실제 급여 시간 분리
- 기록 화면에 **급여 시간** 필드 추가(계획 시간이 기본값, 수정 가능) + **"지금" 버튼**
- 오늘 탭 완료 카드는 계획 시간 대신 실제 급여 시간 표시
- 계획-기록 매칭은 끼니 이름(label) 기준이라 시간 변경해도 안 깨짐

### PR #3 — 바로 기록
- 오늘 탭에 **"+ 바로 기록"** 버튼: 식단표 없이 기록 화면 진입 (현재 시각 + 가장 가까운 mealSlot 이름 자동 제안, 끼니 이름 칩 선택)
- "이 기록을 식단표에도 추가" 옵션 → 식단표 끼니에 `fromRecord: true` 플래그 + 일간/주간/월간 뷰에 **[바로기록] 배지**
- 계획과 매칭 안 되는 기록도 오늘 탭에 완료 카드로 표시(실제 시간순 정렬)
- 재료 0개면 저장 버튼 비활성화

### PR #4 — 기록 → 식단표 동기화 확장
- 식단표 반영 옵션을 **모든 기록 화면**에 노출:
  - 같은 이름 끼니가 식단표에 있으면 → **"이 기록대로 식단표 끼니 업데이트"** (재료 교체, 계획 시간 유지)
  - 없으면 → "이 기록을 식단표에도 추가" (신규 생성)
- 두 경우 모두 fromRecord 배지. 급여표의 계획 대비 비교는 저장 시점 `planSnapshot` 기준이라 원래 계획과의 비교 유지

### PR #5 — 재고 차감 정확도
- **버그 수정**: 재고에 없는/부족한 재료도 전량 차감된 것처럼 알림에 표시 → 실제 차감량만 표시하고 부족분은 "재고 부족(기록만 저장): ○○"로 구분
- `deductFrozen/deductFridge`가 실제 차감량을 반환하도록 변경
- **버그 수정**: 기록 항목에 실제 차감량 `deductedQty` 저장. 재고 부족 상태로 저장한 기록을 수정할 때 기록 수량 전체가 복원되어 재고가 부풀던 문제 해결 (구버전 기록은 qty로 폴백 — 호환 유지)
- 참고: 기록 수정 시 이중 차감 없음 — LOG_SAVE가 기존 차감을 복원 후 재차감하는 구조

## 5. 데이터 모델 메모 (이번 세션에서 추가된 필드)

| 위치 | 필드 | 의미 |
|---|---|---|
| plans[date][].meal | `fromRecord?: true` | 급여 기록에서 생성/동기화된 끼니 (배지 표시용) |
| logs[date][].items[] | `deductedQty` | 실제 재고에서 차감된 양 (냉동=큐브, 냉장=g). 없으면 qty로 폴백 |
| logs[date][] | `time` | 실제 급여 시간 (계획 시간과 별개) |
| logs[date][] | `planSnapshot` | 저장 시점 계획 스냅샷 (계획 대비 비교용, 바로 기록은 null) |

마이그레이션은 `migrateState()`에서 처리. 새 필드는 모두 "없으면 폴백" 방식이라 기존 데이터와 호환됨.

## 6. 알려진 엣지 케이스 / 다음 작업 후보

- 같은 이름(예: 점심)의 기록이 하루에 2개면: 오늘 탭에는 둘 다 보이지만, 급여표 상세 비교 화면은 첫 기록 기준 (권장하지 않는 사용 패턴)
- **개선 후보(제안만 됨)**: 바로 기록에서 식단표에 있는 끼니 이름을 고르면 계획된 재료를 미리 채워주기
- `restoreFrozen/restoreFridge`는 해당 재료의 배치가 모두 삭제된 경우 복원 불가 (기존 한계)
- 소개 이미지 문구/구성 변경 요청 가능성 있음 (3절 참고)

## 7. 이어서 작업하는 방법

```bash
npm install          # 의존성
npx vite             # 개발 서버 (?demo 붙여서 로그인 없이 확인)
npx vite build       # 커밋 전 빌드 확인 (이번 세션 관례)
```

- 새 작업은 main에서 `claude/babycube-app-dev-*` 브랜치를 따서 진행 → PR → 머지(=배포)
- UI 변경은 데모 모드 + Playwright로 실제 동작을 확인한 뒤 커밋하는 것을 관례로 함
- PWA라서 배포 후 사용자 기기에는 앱 완전 종료 후 재실행해야 반영됨
